# Reference + Scope — Theory

**Status:** design / not implemented
**Scope:** the IntelliJ Platform concepts that underpin Block B (identifier resolution + scope) of [`../functionalities.md`](../functionalities.md)
**Plugin module:** `plugin-bbk/`
**Prerequisite:** Block A — see [`../basic-autocomplete/theory.md`](../basic-autocomplete/theory.md)

---

## 1. The central question

Block A teaches the plugin "what keywords does the language have". Block B teaches it something deeper:

> Given the cursor on an identifier `customerName`, what declaration in the program does this name refer to?

That single question opens up everything else: once you know what a name refers to, you get **for free** go-to-declaration, find usages, rename, the "unresolved symbol" error, and an autocomplete that suggests real user identifiers (not just keywords).

The IntelliJ framework for this is called **PsiReference**.

---

## 2. PsiReference — the contract

Every IDENT that *uses* something (not the declaration itself, but a reference to a declaration) gets a `PsiReference` object. It has two key methods:

```
PsiReference {
  PsiElement resolve();    // what do I point to?
  Object[] getVariants();  // if you don't know yet, what options would I have here?
}
```

- **`resolve()`** returns the PSI element of the declaration. If it finds nothing, it returns `null` → IntelliJ marks the IDENT as a "broken reference" (the classic "cannot resolve" red squiggly).
- **`getVariants()`** returns the names valid at this point. This is **what Block A asks Block B for** to enrich autocomplete with user identifiers.

In short: the same machinery yields both resolution and suggestions.

---

## 3. PsiNamedElement — the other side of the contract

For `resolve()` to point at a declaration, the declaration has to be **identifiable by name**. On the PSI side, the declaration implements `PsiNamedElement`:

```
PsiNamedElement {
  String getName();
  PsiElement setName(String newName);   // used by Rename
}
```

For example, `BbkVariableDeclaration` has to be able to answer "my name is `customerName`". Since PSI classes are auto-generated by Grammar-Kit, the way to extend them is via the BNF: declare a `psiImplUtilClass` header and route each declaration rule to a small Java method (`getName`, `setName`, `getNameIdentifier`). Grammar-Kit weaves these into the generated PSI without manual editing.

---

## 4. Scope — "what names can I see from here?"

`PsiReference.resolve()` needs to say "look up the name `X` starting from the cursor position". That requires a **scope**: the ordered list of declarations visible from a code point.

BBK has a scope hierarchy:

```
file
  └── module                  ← top-level: DCL-S, DCL-C, DCL-DS, DCL-PR, DCL-PROC, DCL-F
        └── procedure         ← params + locals of the current DCL-PROC
              └── block       ← variables declared in nested blocks (inline for-init, etc.)
                    └── ...
```

When the cursor is in `block`, visible names are: this block + its ancestors up to the module. Classic lexical scoping.

Conceptually, `BbkScope` is:

```
interface BbkScope {
  List<PsiNamedElement> getDeclarations();
  BbkScope getParent();
}
```

And `BbkScopeWalker.scopeAt(position)` builds that chain by walking PSI ancestors.

---

## 5. Why scope is separated from reference

It's tempting to do everything inside `resolve()`: given an IDENT, walk ancestors looking for declarations with that name. But there are two reasons to separate:

1. **Reuse.** The same scope is used both to resolve (a specific name) and to list variants (every name). If they're entangled, autocomplete has to call `resolve()` N times.
2. **Scope rules orthogonal to references.** Rules like "a variable is not visible before its declaration" or "params are only visible inside their proc" live naturally in the scope. The reference just asks the scope.

It's the classic split: **resolver = engine / scope = knowledge base**.

---

## 6. The resolution chain

From the cursor to a declaration, the flow is:

```
1. User presses Ctrl+B on `customerName`
2. IntelliJ finds the PsiElement under the cursor (a leaf IDENT)
3. Wraps it with the PsiReference I provided (via ReferenceContributor)
4. Calls reference.resolve()
5. My resolve():
     - obtains the searched name: "customerName"
     - builds scopeAt(position)
     - iterates scopes from innermost to outermost
     - in each scope, looks for a PsiNamedElement with that name
     - returns the first match
6. IntelliJ receives the declaration's PsiElement and opens the editor on its line
```

For find-usages, the flow is inverted: given a declaration, IntelliJ scans every IDENT in the file and keeps the ones whose `resolve()` points back at this declaration.

---

## 7. Member access — `obj.field`

Special case: the IDENT after the dot **is not resolved in the ambient scope**. It's resolved in the **scope of the type of `obj`**.

That needs an intermediate step:

```
1. Cursor at `employee.firstName`, on `firstName`
2. The reference of `firstName` is a BbkMemberReference
3. resolve():
     - obtains the LHS: `employee`
     - resolves `employee` (recursively, via BbkIdentReference)
     - obtains its declared type: e.g. BbkDataStructureDeclaration `employeeDs` (via LIKEDS or direct)
     - looks for the subfield `firstName` inside the subfields of `employeeDs`
     - returns that subfield
```

Here Block B brushes against **type resolution** — but **shallowly**. It only needs to follow LIKEDS chains until it reaches a DCL-DS. It is not the full type checker (that's Block D).

A sub-question that matters: what about non-qualified DCL-DS? In BBK, without `QUALIFIED`, subfields are visible directly without the `obj.` prefix. That affects the module or procedure scope, not member access. The two cases must be handled separately.

---

## 8. Subroutines and other non-variable references

Not every IDENT-use is a variable. Block B also has to cover:

- **`exsr name`** → reference to `begsr name`
- **Procedure calls** → `processOrder(...)` resolves to a `DCL-PROC processOrder` or its `DCL-PR`
- **`LIKEDS(name)`** → reference to a `DCL-DS name` (it's an IDENT inside LIKEDS, not just a string)
- **`LIKE(name)`** → reference to another `DCL-S`
- **`BASED(name)`** → reference to the pointer

Each one is a distinct `PsiReference` because the universe of candidates changes:
- `exsr` only looks at subroutines
- function calls only look at procedures/prototypes
- `LIKEDS` only looks at data structures
- etc.

That's why we split into `BbkIdentReference`, `BbkMemberReference`, `BbkSubroutineReference`, and (in this layer) `BbkTypeReference` for LIKE/LIKEDS/LIKEREC — they're "resolution engines" with distinct vocabularies.

---

## 9. What comes for free

Once reference + scope is implemented, **IntelliJ derives by itself** these features:

| Feature | How it's derived |
|---|---|
| **Go to declaration** | `Ctrl+B` → IntelliJ calls `reference.resolve()` → opens the file at that position |
| **Find usages intra-file** | IntelliJ walks every IDENT in the file and keeps those whose `resolve()` points to the selected declaration |
| **Rename intra-file** | Same as find-usages, then calls `setName(...)` on each |
| **Highlight usages** | When the cursor hovers an identifier, IntelliJ underlines every occurrence using the same mechanism |
| **"Unresolved reference" error** | If `resolve()` returns `null`, IntelliJ paints the IDENT red |
| **Enriched autocomplete** | A new `BbkScopeCompletionProvider` (in Block A's framework) asks `BbkScopeWalker.scopeAt(position)` for visible names and emits them as `LookupElement`s |

Five features from one investment.

---

## 10. Caching

`resolve()` is called **many** times. With a file open, IntelliJ re-resolves every IDENT on every keystroke to keep highlights up to date. Without caching, performance collapses.

IntelliJ provides `ResolveCache.getInstance(project).resolveWithCaching(...)`. You pass the `Resolver` implementation in your reference; IntelliJ takes care of invalidating the cache when the PSI changes.

This is not optional complexity — it's **the difference between a usable plugin and one that freezes the editor on 500-line files**. Introduce it from day one.

---

## 11. Limitations of Block B (what B does not solve)

| Limitation | Block that covers it |
|---|---|
| **Cross-file.** An IDENT referring to a proc declared in another `.bbk` stays "unresolved" | C — stubs + index allow resolution without re-parsing the other file |
| **Inferred type.** Knowing that `total = price * vat` yields `PACKED(11:4)` and validating it | D — type system |
| **BIF signatures.** `substr(s, 1, 5)` resolves to "the BIF substr" as an abstract element, but without a signature | E — builtins registry |
| **Cross-file refactor** (rename that updates other files) | C |
| **Member access on arrays or function results** (`employees[i].firstName`, `getCustomer().name`) | Partially possible in B, but the full case needs D's type inference |

---

## 12. Interaction with Block A

Block A stays **intact**. The only change is the addition of **extra providers** (`BbkScopeCompletionProvider`, `BbkMemberCompletionProvider`) that coexist with the 14 already in place. The framework is designed for this: extend, don't replace.

Where Block A *is* refined is in the gating imperfections that surfaced during sandbox testing: now we can check "does an `if` actually exist before this `else`?" by walking PSI with structural knowledge, instead of relying on patterns alone.

## 13. Minimal class set for Block B

Putting the theory into a concrete file map:

```
plugin-bbk/src/main/java/com/larena/boxbreaker/plugin/bbk/
├── psi/util/
│   └── BbkPsiImplUtil.java                 — getName/setName/getNameIdentifier for every named declaration
├── reference/
│   ├── BbkReferenceContributor.java        — registers which IDENTs receive references
│   ├── BbkIdentReference.java              — generic IDENT → declaration resolution
│   ├── BbkMemberReference.java             — `obj.field` resolution via LHS type
│   ├── BbkSubroutineReference.java         — `exsr name` → `begsr name`
│   └── BbkTypeReference.java               — LIKE/LIKEDS/LIKEREC IDENT → DCL-S / DCL-DS
├── scope/
│   ├── BbkScope.java                       — interface: getDeclarations() + getParent()
│   ├── BbkScopeWalker.java                 — scopeAt(position) static API
│   ├── BbkModuleScope.java                 — top-level declarations
│   ├── BbkProcedureScope.java              — params + locals of the enclosing DCL-PROC
│   └── BbkBlockScope.java                  — locals declared in nested blocks (e.g. for-init)
├── types/
│   └── BbkTypeResolver.java                — shallow type lookup (LIKEDS chain → DCL-DS)
└── completion/providers/
    ├── BbkScopeCompletionProvider.java     — emits identifiers from scope as completions
    └── BbkMemberCompletionProvider.java    — emits subfields after `.`
```

Plus declarations in `BBK.bnf` (`psiImplUtilClass` header + `mixin` attributes on each named declaration rule) and a `<psi.referenceContributor>` extension in `plugin.xml`.

---

## 14. What Block B does NOT cover (intentionally)

To stay honest about scope, Block B leaves these out:

- **Cross-file resolution** — see Block C.
- **Type checking / smart completion** — see Block D.
- **BIF signatures and inline docs** — see Block E.
- **Implicit subfield exposure** for nested `LIKEDS` chains beyond two levels — limited by the shallow type resolver; full graph support belongs to D.
- **Polyvariant resolution** (overloaded procedures) — BBK does not allow overloading, so this stays N/A.
